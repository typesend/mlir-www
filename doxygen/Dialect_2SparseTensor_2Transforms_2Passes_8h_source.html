33 /// Defines a parallelization strategy. Any independent loop is a candidate
34 /// for parallelization. The loop is made parallel if (1) allowed by the
35 /// strategy (e.g., AnyStorageOuterLoop considers either a dense or sparse
36 /// outermost loop only), and (2) the generated code is an actual for-loop
37 /// (and not a co-iterating while-loop).

40  kDenseOuterLoop,

43  kAnyStorageAnyLoop

79 /// Defines a strategy for implementing sparse-to-sparse conversion.
80 /// `kAuto` leaves it up to the compiler to automatically determine
81 /// the method used. `kViaCOO` converts the source tensor to COO and
82 /// then converts the COO to the target format. `kDirect` converts
83 /// directly via the algorithm in <https://arxiv.org/abs/2001.02609>;
84 /// however, beware that there are many formats not supported by this
85 /// conversion method.
86 ///
87 /// The presence of the `kAuto` option violates our usual goal of keeping
88 /// policy completely separated from mechanism. The reason it exists is
89 /// because (at present) this strategy can only be specified on a per-file
90 /// basis. To see why this is a problem, note that `kDirect` cannot
91 /// support certain conversions; so if there is no `kAuto` setting,
92 /// then whenever a file contains a single non-`kDirect`-able conversion
93 /// the user would be forced to use `kViaCOO` for all conversions in
94 /// that file! In the future, instead of using this enum as a `Pass`
95 /// option, we could instead move it to being an attribute on the
96 /// conversion op; at which point `kAuto` would no longer be necessary.
97 enum class SparseToSparseConversionStrategy { kAuto, kViaCOO, kDirect };
